# Deployment Guide — Averias

> **Averias** is a Django-based glossary of equipment faults. This guide covers deploying it on an AWS EC2 instance alongside other Dockerized applications, with NGINX already configured as a reverse proxy.

## 1. Prerequisites

Before starting, ensure you have:

- **AWS EC2 instance** (Ubuntu 22.04+ recommended) running and accessible via SSH
- **Domain name** pointing to the instance's public IP (optional but recommended)
- **Docker** already installed on the instance (the instance runs other Dockerized applications)
- **NGINX** already installed and configured on the instance (it coexists with other apps)
- **UFW** already configured (ports 22, 80, 443 open)
- **GitHub account** with access to [GitHub Container Registry (GHCR)](https://ghcr.io)
- **Docker** installed on your local machine (for building images)

### Required environment variables

To be able to deploy, you will need the following values ready:

| Variable | Description |
|---|---|
| `ALLOWED_HOSTS` | Comma-separated domain/IPs (e.g., `averias.example.com,www.averias.example.com`) |
| `CSRF_TRUSTED_ORIGINS` | Comma-separated origins (e.g., `https://averias.example.com`) |
| `DB_NAME` | MariaDB database name |
| `DB_USER` | MariaDB database user |
| `DB_PASSWORD` | MariaDB database password |
| `DB_ROOT_PASSWORD` | MariaDB root password (optional; defaults to `DB_PASSWORD` if not set) |
| `DB_HOST` | Database host (default: `averias-db` — the MariaDB container name) |
| `DB_PORT` | Database port (default: `3306`) |
| `SECRET_KEY` | Django secret key (auto-generated by `setup-env.sh`) |
| `DEBUG` | `True` or `False` (default: `False`) |
| `SECURE_SSL_REDIRECT` | `True` or `False` (default: `True`) |
| `SECURE_HSTS_SECONDS` | HSTS max-age in seconds (default: `31536000`) |
| `SESSION_COOKIE_AGE` | Session lifetime in seconds (default: `28800`) |
| `SESSION_COOKIE_SECURE` | `True` or `False` (default: `True`). Set to `False` for HTTP-only testing. |
| `CSRF_COOKIE_SECURE` | `True` or `False` (default: `True`). Set to `False` for HTTP-only testing. |
| `USE_S3` | Whether to serve static files from S3 (`True`/`False`) |
| `AWS_ACCESS_KEY_ID` | (If `USE_S3=True` and **no** IAM Role) AWS access key |
| `AWS_SECRET_ACCESS_KEY` | (If `USE_S3=True` and **no** IAM Role) AWS secret key |
| `AWS_STORAGE_BUCKET_NAME` | (If `USE_S3=True`) S3 bucket name |
| `AWS_S3_REGION_NAME` | (If `USE_S3=True`) S3 region (default: `eu-west-1`) |
| `AWS_S3_CUSTOM_DOMAIN` | (If `USE_S3=True`) Custom S3 endpoint (default: `<bucket>.s3.<region>.amazonaws.com`). Required for regions that need explicit endpoint. |

## 2. Environment Setup

Run the interactive script on the EC2 instance to generate `~/.averias.env`:

```bash
./setup-env.sh
```

This will prompt you for:

1. **Django settings** — `ALLOWED_HOSTS`, `DEBUG`, `CSRF_TRUSTED_ORIGINS`, `SECRET_KEY` (auto-generated)
2. **Database** — `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_ROOT_PASSWORD`, `DB_HOST`, `DB_PORT`
3. **AWS S3 (optional)** — `USE_S3`, and if enabled:
   - Asks if you use an **IAM Role** (recommended) — if yes, skips `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`
   - Otherwise, prompts for access keys
   - Always prompts for `AWS_STORAGE_BUCKET_NAME`, `AWS_S3_REGION_NAME`, and `AWS_S3_CUSTOM_DOMAIN`

> **Tip:** For production, set `DB_HOST=averias-db` (the name of the MariaDB container, as both containers share a Docker network). The `.averias.env` file is automatically secured with restricted permissions (`chmod 600`).

Alternatively, you can create `~/.averias.env` manually with the required variables (see `setup-env.sh` for the expected format).

## 3. Docker Image

### 3.1 Build the image

From your local machine (or CI/CD pipeline):

```bash
docker build -t ghcr.io/psotsan/averias:latest .
```

### 3.2 Push to GitHub Container Registry

```bash
# Log in to GHCR
echo "$GITHUB_PAT" | docker login ghcr.io -u <your-username> --password-stdin

# Push the image
docker push ghcr.io/psotsan/averias:latest
```

> Replace `<your-username>` and `psotsan` with your actual GitHub username and organization.

### 3.3 Tag versions (optional)

For production, tag specific versions:

```bash
docker tag ghcr.io/psotsan/averias:latest ghcr.io/psotsan/averias:v1.0.0
docker push ghcr.io/psotsan/averias:v1.0.0
```

## 4. Deployment

`deploy.sh` manages two Docker containers connected through a shared network:

| Container | Name | Image | Purpose |
|---|---|---|---|
| Database | `averias-db` | `mariadb:11.4` | MariaDB server |
| App | `averias` | `ghcr.io/psotsan/averias:latest` | Django application |

### 4.1 Run the deploy script

Make sure `~/.averias.env` is in place, then run:

```bash
./deploy.sh
```

The script does the following:

1. **Preflight** — checks that `.env` exists and Docker is installed
2. **Network** — creates a Docker network `averias-net` (if it doesn't exist)
3. **Database** — starts a MariaDB container (`averias-db`) with:
   - Persistent volume (`averias-db-data`) for data durability
   - Auto-restart policy (`unless-stopped`)
   - Database and user created from `.env` variables
4. **App** — pulls the image from GHCR and starts the web container (`averias`) with:
   - Port mapping: `8001:8001` (host:container)
   - Environment variables loaded from `~/.averias.env`
   - Auto-restart policy (`unless-stopped`)
5. **Wait** — waits for both containers to be ready (MariaDB ping + Django setup)
6. **Migrations** — runs `python manage.py migrate`
7. **Superuser** — creates a Django admin user (skips if already exists)
8. **Collectstatic** — collects static files
9. **Smoke test** — verifies the app responds on `http://localhost:8001`

### 4.2 Custom deployment options

```bash
# Use a different image tag
./deploy.sh --tag ghcr.io/psotsan/averias:v1.0.0

# Use a different port
./deploy.sh --port 8080

# Custom container names
./deploy.sh --container-name averias-prod --db-container-name averias-prod-db

# Custom network name
./deploy.sh --network-name my-custom-net

# Skip migrations
./deploy.sh --skip-migrations

# Skip collectstatic
./deploy.sh --skip-collectstatic

# Custom .env file path
./deploy.sh --env-file /path/to/custom/.env
```

### 4.3 DNS setup

Create a **DNS A record** pointing to your instance IP:

| Type | Name | Value |
|---|---|---|
| **A** | `averias` | `<instance-public-ip>` |

> ⚠️ Use an **A record**, not a CNAME. A CNAME cannot point to an IP address and will cause certbot to fail with `NXDOMAIN`.

### 4.4 NGINX configuration

Create a server block for the subdomain at `/etc/nginx/sites-available/averias`:

```nginx
server {
    listen 80;
    server_name averias.example.com;
    return 301 https://$server_name$request_uri;
}

server {
    server_name averias.example.com;

    location /static/ {
        return 301 https://<your-bucket>.s3.<region>.amazonaws.com/;
    }

    location / {
        include proxy_params;
        proxy_pass http://127.0.0.1:8001;
    }

    listen 443 ssl; # managed by Certbot
    ssl_certificate /etc/letsencrypt/live/psotorrio.click/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/psotorrio.click/privkey.pem;
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;
}
```

Enable it and obtain the SSL certificate:

```bash
sudo ln -s /etc/nginx/sites-available/averias /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx
sudo certbot --nginx -d averias.example.com
```

> Certbot will add the new domain to your existing certificate automatically if it shares the same root domain.

### 4.5 Docker Compose (local/development)

For local development, use `docker-compose.yml`:

```bash
docker compose up -d --build
```

This starts both a MariaDB container (`db`) and the web application (`web`) with hot-reload via volume mounts. This setup is equivalent to the production deployment but with the code mounted directly for development.

## 5. Troubleshooting

### 5.1 MariaDB container fails with "password option is not specified"

**Cause:** The `.env` file contains special characters (`!`, `$`, etc.) that bash interpreted during generation.

**Solution:** Regenerate the `.env` file (the script now handles special characters safely).

```bash
rm ~/.averias.env
./setup-env.sh
```

### 5.2 App returns 400 Bad Request

**Cause:** `SECURE_SSL_REDIRECT=True` and `SESSION_COOKIE_SECURE=True` but the request is HTTP.

**Solutions:**
- Use HTTPS via the configured subdomain
- Or for testing, set these to `False` in `~/.averias.env` and restart:
  ```
  SECURE_SSL_REDIRECT=False
  SESSION_COOKIE_SECURE=False
  CSRF_COOKIE_SECURE=False
  ```

### 5.3 App returns 500 Internal Server Error

Check Gunicorn/Django logs:

```bash
docker logs averias --tail 40
```

Common causes:
- Pending migrations: `docker exec averias python manage.py migrate`
- Database connection issues: verify `DB_HOST` is `averias-db`
- Missing environment variables: verify `~/.averias.env` has all required values

### 5.4 Gunicorn fails with "Permission denied: '/nonexistent'"

**Cause:** The container runs as `nobody` user but `$HOME` is not set.

**Solution:** The Dockerfile now sets `ENV HOME=/home/nobody`. Rebuild and push the image:

```bash
docker build -t ghcr.io/psotsan/averias:latest .
docker push ghcr.io/psotsan/averias:latest
```

## 6. Post-Deployment

### 6.1 Load test data (optional)

If you want to seed the database with sample equipment types, models, and faults:

```bash
docker exec averias python cargar_datos.py
```

This creates:
- **2 equipment types** (Scanner, Tissue Processor)
- **6 models**
- **48 faults** with random reference codes

### 6.2 Verify the app is running

```bash
# Smoke test (already run by deploy.sh)
curl -I http://localhost:8001

# Check container logs
docker logs averias

# Check all running containers
docker ps --filter network=averias-net
```

### 6.3 Create a Django superuser (manual)

If the automated step failed or you need additional users:

```bash
docker exec -it averias python manage.py createsuperuser
```

### 6.4 Database backup and restore

```bash
# Backup
docker exec averias-db mariadb-dump -u root -p"$DB_PASSWORD" "$DB_NAME" > backup.sql

# Restore
docker exec -i averias-db mariadb -u root -p"$DB_PASSWORD" "$DB_NAME" < backup.sql
```

### 6.5 Useful commands

```bash
# View app logs
docker logs -f averias

# View database logs
docker logs averias-db

# Access the database
docker exec -it averias-db mariadb -u root -p

# Run a Django management command
docker exec -it averias python manage.py <command>

# Restart a container
docker restart averias

# Stop everything
docker stop averias averias-db

# Remove everything (data is persisted in volumes)
docker rm -f averias averias-db
docker network rm averias-net
# To also delete the data volume:
docker volume rm averias-db-data
```

## 7. Local Development

### 7.1 Quick start

```bash
# Clone the repository
git clone <repo-url> && cd fault_knowledge_base

# Start with Docker Compose
docker compose up -d --build

# Or run locally (without Docker)
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
python manage.py runserver 0.0.0.0:8001
```

### 7.2 Environment variables for development

Create a `.env` file at the project root:

```bash
SECRET_KEY='dev-secret-key-insecure'
DEBUG=True
ALLOWED_HOSTS=localhost,127.0.0.1
CSRF_TRUSTED_ORIGINS=http://localhost:8001
DB_NAME=averias
DB_USER=averias_user
DB_PASSWORD=averias_pass
DB_HOST=db
DB_PORT=3306
USE_S3=False
```

### 7.3 Useful commands

```bash
# Run migrations
docker compose exec web python manage.py migrate

# Collect static files
docker compose exec web python manage.py collectstatic

# Create superuser
docker compose exec web python manage.py createsuperuser

# View logs
docker compose logs -f web

# Load test data
docker compose exec web python cargar_datos.py
```
